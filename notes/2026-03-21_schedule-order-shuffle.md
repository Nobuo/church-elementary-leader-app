# スケジュール処理順のシャッフルで結果に揺らぎを出す

## 日付: 2026-03-21

## 要望

割り当て回数が「階段状」になるのが気になる。毎回生成するたびに違う組み合わせになるよう、乱数を使って計算の順番を変化させたい。ただし優先度は変えない。

## 背景

### 既に対応済みの施策

- **シャッフルタイブレーク（タスク017）**: 同スコア時のペア選択をランダム化
- **プール内相対均等配分（タスク022）**: 均等配分の基準をプール内minに変更

### まだ残る階段パターンの原因

現在の `generateAssignments()` はスケジュールを**日付順に処理**する:

```typescript
const activeDates = schedules.filter(s => !s.isExcluded).sort((a, b) => a.date.localeCompare(b.date));
for (const schedule of activeDates) { ... }
```

日付順固定のため、**最初の日に選ばれたメンバーのcountが+1され、以降の日で不利になる**というカスケードが毎回同じパターンで起きる。シャッフルタイブレークがあっても、count差（+50/回）がタイブレーク（±5程度）を大きく上回るため、結果は似たパターンに収束する。

### 具体例

4月の4日間を処理する場合:
- 日付順: 4/5→4/12→4/19→4/26
- 4/5でAさんが選ばれる → Aさんのcount+1 → 4/12以降でAさんは不利
- **毎回同じ順序で処理するため、同じカスケードが起きやすい**

## 改善案

### スケジュール処理順をシャッフルする

```typescript
// 変更前
const activeDates = schedules.filter(s => !s.isExcluded).sort((a, b) => a.date.localeCompare(b.date));

// 変更後
const activeDates = shuffle(schedules.filter(s => !s.isExcluded));
```

- 4/26→4/12→4/5→4/19 のような順序で処理
- 最初に選ばれるメンバーが毎回変わる → カスケードパターンが変化 → 結果に揺らぎ
- **最終的に生成されたAssignmentはscheduleIdで紐づくため、表示は日付順のまま**

### なぜこれが有効か

- count差ペナルティ（+50/回）は非常に強い → 処理順が結果を大きく左右する
- 1日目で count=0→1 になったメンバーは、2日目以降 (1-0)*50=+50 のペナルティを受ける
- 処理順が変わると「誰が最初に+1されるか」が変わり、全体の分布が変化する
- 優先度（ハード制約、BOTH温存、ヘルパー後回し等）は一切変わらない

### 注意点

- 「月内重複」ペナルティ（+100）は処理順に依存するが、これはそもそも月内で2回以上の割り当てを抑制するものなので、処理順が変わっても効果は保たれる
- 結果の割り当ては scheduleId に紐づくため、表示順に影響なし
- 既存テストでシャッフル前提で書かれていないものは修正不要（結果はランダムだが制約は遵守）

## 期待される効果

- 毎回生成するたびに異なる組み合わせが出る
- 階段パターンが崩れ、より「ランダム」な印象になる
- 優先度・制約は全て維持
