# 仕様書: 日付指定メンバーの優先割り当て

## 機能概要

`availableDates`（参加可能日）が設定されているメンバーを、その日に優先的に割り当てる。日付指定は「その日にしか来られない」ことを意味するため、全日参加可能なメンバーよりも優先して割り当てることで、日付指定メンバーの割り当て機会を確保し、全体の公平性を向上させる。

## 背景

現在のアルゴリズムでは、`availableDates` を持つメンバーは「その日に参加可能かどうか」のフィルタリングにのみ使用されている（`member.isAvailableOn(dateStr)`）。候補選出後のスコアリングでは全メンバーが同等に扱われるため、日付指定メンバーが他の日に回されるリスクがない一方で、「その日しか来られない」メンバーが選ばれずに終わる可能性がある。

## ドメインモデル

### 変更対象

| 対象 | 変更内容 |
|------|---------|
| `assignment-generator.ts` / `scorePair()` | 日付指定メンバーに対するスコアボーナス（減点）を追加 |

### 新規追加なし

エンティティ・値オブジェクト・リポジトリの変更は不要。既存の `Member.availableDates` プロパティを活用する。

## ビジネスルール

### 優先度の定義

`availableDates` が設定されているメンバー（＝日付制限あり）は、設定されていないメンバー（＝全日参加可能）よりもスコアリングで優遇する。

### スコアリング変更

`scorePair()` 内で、各メンバーに対して以下のボーナスを適用:

| 条件 | スコア変更 |
|------|-----------|
| `member.availableDates` が非null・非空配列（日付指定あり） | **-30** |
| `member.availableDates` が null/undefined/空配列（全日参加可能） | 0（変更なし） |

### ボーナス値の根拠

- 月内重複ペナルティ（100）より低く、ペア多様性ペナルティ（10）より高い
- 夫婦回避ペナルティ（30）と同等
- これにより、日付指定メンバーは優先されるが、必須制約（言語バランス: 100,000、同性ペア: 100,000）やその他の強い制約を覆さない

### 制約との関係

| 制約 | 影響 |
|------|------|
| 必須: 参加可能日 | 影響なし（フィルタリング段階で処理済み） |
| 必須: 言語バランス | 影響なし（スコア100,000で不可） |
| 強: 同性ペア制限 | 影響なし（スコア100,000で不可） |
| 弱: 月内重複回避 | 日付指定メンバーが月内重複しても選ばれやすくなる（-30 vs +100で、重複ペナルティが上回る） |
| 弱: 均等な担当回数 | 日付指定メンバーの担当回数が多くても優先される場合がある（-30 vs 回数差×50） |
| 弱: 夫婦回避 | 同等のペナルティ（±30）のため、日付指定で夫婦回避が打ち消される可能性あり |
| 弱: ペア多様性 | 日付指定ボーナスが上回る（-30 vs +10） |

### 日付指定メンバー同士の競合

日付指定メンバーが同一日に複数いる場合、両者に同じボーナスが適用されるため、既存のスコアリング（担当回数、ペア多様性等）で自然に選出される。

## ユースケース

### UC-1: 正常系 — 日付指定メンバーの優先割り当て

**前提条件:**
- メンバーA: `availableDates = ["2027-04-06", "2027-04-13"]`（2日のみ参加可能）
- メンバーB: `availableDates = null`（全日参加可能）
- メンバーA, Bは同じ `gradeGroup`

**動作:**
- 2027-04-06 の割り当て時、メンバーAがメンバーBより優先的に選出される
- メンバーBは他の日曜日に割り当てられる

### UC-2: 正常系 — 必須制約との共存

**前提条件:**
- メンバーA: 日付指定あり、言語=JAPANESE
- メンバーC: 日付指定なし、言語=BOTH
- グループに英語対応者が必要

**動作:**
- メンバーAの日付指定ボーナス(-30)があっても、言語バランス違反ペナルティ(+100,000)が上回り、メンバーCが選出される
- 言語バランスが問題ないペアリングであれば、メンバーAが優先される

### UC-3: 異常系 — 日付指定メンバーが多すぎる

**前提条件:**
- 高学年担当で日付指定ありのメンバーが3名以上、同じ日に参加可能

**動作:**
- 最大2名（Group1 + Group2 に各1名）まで選出される
- 残りは選出されない（担当回数が増えない）
- 警告は発生しない（正常な振る舞い）

### UC-4: 正常系 — 日付指定なしメンバーのみ

**前提条件:**
- 全メンバーが `availableDates = null`

**動作:**
- ボーナスが誰にも適用されないため、現行と完全に同じ動作

for (const m of [member1, member2]) {
