# 仕様書: スケジュール処理順シャッフル

## 機能概要

`generateAssignments()` のスケジュール処理順を日付順固定からランダム順に変更する。処理順のカスケード効果により毎回異なる割り当て結果を生み出し、「階段状」分布を解消する。

### 背景

- スケジュールが日付順に処理されるため、最初の日に選ばれたメンバーが以降の日で count+1 のペナルティを受ける
- このカスケードが毎回同じパターンで起き、結果が似たような階段状分布に収束する
- シャッフルタイブレーク（タスク017）やプール内相対均等配分（タスク022）を導入済みだが、count差ペナルティ（+50/回）がタイブレーク（±5程度）を大きく上回るため、処理順の影響が支配的

## ビジネスルール

### R1: スケジュール処理順のシャッフル

```
変更前:
  activeDates = schedules.filter(非除外).sort(日付順)

変更後:
  activeDates = shuffle(schedules.filter(非除外))
```

既存の `shuffle()` 関数（Fisher-Yates）を再利用する。

### R2: 結果の表示順は変更なし

生成された Assignment は `scheduleId` でスケジュールに紐づく。アプリケーション層・プレゼンテーション層は `scheduleId` 経由で日付を取得するため、処理順に関係なく日付順で表示される。

### R3: 全ての既存制約・優先度は維持

- ハード制約（言語バランス、同性制約、CLASS_LANGUAGE_COVERAGE）
- ソフト制約（BOTH温存、ヘルパー後回し、配偶者回避、ペア多様性、均等配分）
- 月内重複ペナルティ（+100）
- イベント日のヘルパー除外

全て従来通り。変わるのは「どの日を先に処理するか」のみ。

### R4: 月内重複ペナルティとの関係

月内重複ペナルティ（+100）は `monthAssignments` を参照する。処理順が変わると、ある日の処理時点で `monthAssignments` に含まれる既処理日の割り当てが異なる。しかしこれは正常な動作 — 重複ペナルティは「既に割り当てられたメンバーを避ける」ためのものであり、処理順によらず効果を発揮する。

## ユースケース

### UC1: 通常の月次生成

**前提:** 4月の日曜4日間、メンバー23人

**変更前:** 毎回 4/5→4/12→4/19→4/26 の順で処理 → 同じカスケード → 似た結果

**変更後:** ランダム順（例: 4/19→4/5→4/26→4/12）→ 異なるカスケード → 異なる結果

**結果:** 生成ボタンを押すたびに異なる組み合わせが出る。制約は全て遵守。

### UC2: 連続して2回生成

**前提:** 同じ月に対して2回生成する

**処理:**
- 1回目: 4/26→4/12→4/5→4/19 の順で処理
- 2回目: 4/5→4/19→4/26→4/12 の順で処理

**結果:** 異なる割り当て結果が得られる。ユーザーは気に入った方を採用できる。
